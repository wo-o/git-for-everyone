# git-workflow 가이드

git-workflow는 GitHub Flow라는 작업 방식을 안내해주는 플러그인입니다.

코드를 수정할 때 main 브랜치에서 바로 작업하는 대신, 별도의 브랜치를 만들어서 작업하고, 검토를 받은 후 합치는 안전한 흐름을 따르도록 도와줍니다.

## 먼저 알아야 할 용어

- **브랜치(branch):** 코드의 "복사본"이라고 생각하세요. 원본(main)을 건드리지 않고 따로 작업할 수 있는 공간입니다.
- **main 브랜치:** 프로젝트의 "완성본"이 들어있는 기본 브랜치입니다. 항상 정상 동작하는 코드만 여기에 있어야 합니다.
- **feature 브랜치:** 새 기능이나 수정을 위해 main에서 만든 작업용 브랜치입니다. 이름은 보통 `feat/로그인`, `fix/버그수정`처럼 짓습니다.
- **커밋(commit):** 작업 내용을 저장하는 것입니다. 게임의 "세이브 포인트"와 비슷합니다.
- **push:** 내 컴퓨터의 커밋을 GitHub에 올리는 것입니다.
- **PR (Pull Request):** "이 변경사항을 main에 합쳐주세요"라는 요청입니다. 다른 사람이 코드를 검토할 수 있습니다.

## 이 플러그인은 어떻게 동작하나요?

두 가지 기능이 있습니다:

1. **자동 차단:** main/master 브랜치에서 commit이나 merge를 하려고 하면 막아줍니다.
2. **다음 단계 안내:** `/git-workflow:workflow`를 실행하면 지금 상태에서 뭘 해야 하는지 알려줍니다.

## 무엇을 막아주나요?

### main/master 브랜치에서 commit 차단

main 브랜치에서 직접 commit하면 코드 리뷰 없이 완성본이 바뀌게 됩니다.
실수가 있어도 아무도 잡아주지 못합니다.

| 상황 | 결과 | 이유 |
|---|---|---|
| main 브랜치에서 `git commit` | 차단 | 리뷰 없이 완성본이 바뀜 |
| master 브랜치에서 `git commit` | 차단 | 위와 같음 (master는 main의 다른 이름) |
| feat/login 브랜치에서 `git commit` | 허용 | feature 브랜치에서는 자유롭게 작업 가능 |

### main/master 브랜치에서 merge 차단

merge는 다른 브랜치의 코드를 현재 브랜치에 합치는 것입니다.
main에서 merge하면 PR 없이 코드가 합쳐져서 리뷰를 건너뛰게 됩니다.

| 상황 | 결과 | 이유 |
|---|---|---|
| main에서 `git merge feat/login` | 차단 | PR을 통해 합쳐야 안전함 |
| feat/login에서 `git merge main` | 허용 | feature 브랜치에 main의 최신 내용을 가져오는 것은 안전 |

### 막지 않는 것들

main 브랜치에서도 아래 작업은 자유롭게 할 수 있습니다:

- `git checkout -b feat/내기능` — 새 브랜치 만들기 (이게 올바른 시작 방법!)
- `git pull` — GitHub에서 최신 코드 가져오기
- `git status` — 현재 상태 확인하기
- `git log` — 커밋 기록 보기
- `git add` — 파일 준비하기 (commit만 안 하면 됨)

## 차단되면 어떻게 보이나요?

```
차단: main 브랜치에서 직접 commit 불가
  → feature 브랜치를 만들고 작업하세요: git checkout -b feat/my-feature
```

차단되면 당황하지 마세요. 대안을 바로 알려주니 그대로 따라하면 됩니다.

## /git-workflow:workflow 사용법

"지금 뭘 해야 하지?" 싶을 때 `/git-workflow:workflow`를 실행하세요.
현재 상태를 자동으로 파악하고 다음 단계를 알려줍니다.

### 상황 1: main 브랜치에 있고, 변경 없음
→ 새 작업을 시작할 준비가 되었습니다.
→ 안내: `git checkout -b feat/기능이름`으로 새 브랜치를 만드세요.

### 상황 2: main 브랜치에 있고, 변경사항이 있음
→ main에서 실수로 작업을 시작한 상황입니다.
→ 안내: 변경사항을 새 브랜치로 안전하게 옮기는 방법을 알려줍니다.

### 상황 3: feature 브랜치에 있고, 저장 안 된 변경이 있음
→ 작업은 했지만 아직 commit(저장)하지 않은 상황입니다.
→ 안내: `git add`와 `git commit`으로 저장하는 방법을 알려줍니다.

### 상황 4: feature 브랜치에 있고, GitHub에 안 올린 커밋이 있음
→ 커밋은 했지만 GitHub에 아직 올리지 않은 상황입니다.
→ 안내: `git push`와 `gh pr create`로 올리고 PR을 만드는 방법을 알려줍니다.

### 상황 5: feature 브랜치에 있고, 모든 게 깨끗한 상태
→ 모든 작업이 GitHub에 올라간 상태입니다.
→ 안내: PR 상태를 확인하거나 새 작업을 시작하도록 안내합니다.

## GitHub Flow란?

GitHub Flow는 코드를 안전하게 관리하는 간단한 규칙입니다:

```
main 브랜치       ●─────────────────●──────── (항상 안전한 코드)
                   \               /
feature 브랜치      ●───●───●───●  (여기서 작업)
                    작업  커밋  push  PR머지
```

1. main에서 새 브랜치를 만든다
2. 그 브랜치에서 자유롭게 작업하고 커밋한다
3. GitHub에 push하고 PR을 만든다
4. 리뷰를 받고 main에 합친다

이 플러그인은 이 흐름을 자연스럽게 따르도록 도와줍니다.

## 자주 묻는 질문

**Q: main에서 commit하고 싶은데 차단됩니다**
A: main에서는 직접 commit하지 않는 것이 좋습니다. 이렇게 해보세요:
  1. 이미 코드를 수정했다면 → `/git-workflow:workflow`를 실행하세요. 변경사항을 새 브랜치로 옮기는 방법을 안내합니다.
  2. 아직 수정 전이라면 → `git checkout -b feat/내기능`으로 먼저 브랜치를 만드세요.

**Q: merge가 뭐고, 왜 main에서 하면 안 되나요?**
A: merge는 두 브랜치의 코드를 합치는 것입니다.
main에서 직접 merge하면 다른 사람의 검토 없이 코드가 합쳐집니다.
대신 PR을 만들면 변경 내용을 보고 검토한 뒤 안전하게 합칠 수 있습니다.

**Q: feature 브랜치에서 main의 최신 코드를 가져오고 싶습니다**
A: feature 브랜치에서 `git merge main`을 하면 됩니다. 이건 차단되지 않습니다.
feature 브랜치에 main의 최신 내용을 가져오는 것은 안전한 작업입니다.

**Q: 다음에 뭘 해야 할지 모르겠습니다**
A: `/git-workflow:workflow`를 실행하세요. 현재 상태를 자동으로 분석해서 다음에 할 일을 알려줍니다.

**Q: 브랜치 이름은 어떻게 짓나요?**
A: 보통 이런 규칙을 따릅니다:
  - 새 기능: `feat/로그인`, `feat/검색기능`
  - 버그 수정: `fix/로그인오류`, `fix/깨진링크`
  - 기타: `chore/정리`, `docs/설명추가`
